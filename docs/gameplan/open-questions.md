- ~~Is the whole Sheet Happens library GP7 (zip) format, or a mix of GP6 (~~~~`.gpx`~~~~) and GP7/8?~~ **Decision made 2026-07-20** — keep BCFS/BCFZ code regardless, pending a full library audit. See Decision Log.
- ~~What do the ~~~~`<Slide><Flags>`~~~~ integer values in GPIF actually mean, bit for bit?~~ **Resolved 2026-07-17** — cross-checked against editor-on-fire's GP importer (GP5+ bitmask 1/2/4/8/16/32). Further confirmed with real flag combination {1,2,4,20,32}. Considered closed.
- Which track is "the lead guitar" isn't always obvious (sometimes clearly named, sometimes 3 tracks all named `Overdriven Guitar`). `dump-ir --track N` punts this to the user for now; still open whether picking a track can be automated (e.g. "most technique flags" or "highest average pitch") or should just always ask.
- Step-size bucketing (`_interval_to_step`: semitone gap → lane-step magnitude 1-4) remains our own unconfirmed heuristic. RBN's "preserve motion over consistency" principle is sourced-confirmed and supports the general direction but doesn't confirm any particular step-sizing algorithm. A hypothesis that real charters force a flat 1-lane-per-note step for detected monotonic runs (3+ notes same direction), reserving proportional sizing for disjunct/leap motion, remains unconfirmed — would need a real chart example. Not implemented.
- ~~Should phrase-boundary reset set the cursor to 0 (green) regardless of the new phrase's direction, or account for which way the new phrase moves?~~ **Resolved 2026-07-21** — always reset to 0 (green), regardless of direction. Considered settled on real-guitar fretting-hand-anchoring reasoning (moderate confidence — sound reasoning, not drawn from a chart-authoring source specifically). No change made.
- ~~Same-tick lane-collision handling for genuine chords: the ~~~~`_assign_group_lanes`~~~~ placement logic for a chord's non-anchor lanes was an unresolved placeholder.~~ **Resolved 2026-07-21 (mechanism)** — placeholder replaced with the scored chord-shape heuristic (`_chord_shape_candidates` + `_rank_chord_shape`); the open-chug-chord xfail was fixed as a structural side effect. See Decision Log.
- **Still open (quality):** does a real power chord's lane placement *read well* on an actual chart? Nearest-free-lane / chord-shape-scored placement is implemented and unit-tested but has never been run against a real chord-bearing `.gp` file. Only a real playtest answers this — no amount of further code reasoning substitutes for it.
- ~~Cursor-resync drift (whether repeatedly resyncing the cursor to a chord's chosen anchor lane could accumulate error over a long progression) was bundled into the chord-collision question above as an open architectural risk.~~ **Eliminated 2026-07-21, not just resolved** — proven both mathematically (the correction is bounded to `[-4,4]` per chord by construction, with no mechanism to bias future step size) and by an encoded regression test (`test_cursor_resync_does_not_drift_over_long_progression`). This is no longer tracked as an open question at all; see Decision Log.
- Closed 2026-07-21, N/A: RBN's Trill/Tremolo lane markers are a Rock Band MIDI-engine-specific mechanic with no equivalent in the `.chart` text format this project emits. No action needed.
- `ghost_note` in `ir_gpif.py` is now `"GhostNote" in props` (inferred from the GPIF property naming pattern), but unverified against a real file that carries a ghost note. If one turns up and `ghost_note` stays `False` despite a visible ghost note in the tab, check whether GP7 uses a different property name/location and fix.
- **Backlog (2026-07-21):** a dedicated `mapper.py` regression corpus (`single_notes/`, `dyads/`, `triads/`, `staircase_runs/`, `repeated_chords/`, `awkward_spreads/` — synthetic IR fixtures paired with expected lane output), suggested during PR #8 review. Not built yet — real payoff starts once weight-tuning against actual playtest feedback begins, which needs the still-open "real chord-bearing file" playtest above.
- **Next-iteration priorities (2026-07-21), per an external review pass, in order:**
	1. Tune `_rank_chord_shape`'s `_WEIGHT_*` constants using a representative corpus instead of intuition. Blocked on a real chord-bearing `.gp` file or a curated synthetic corpus — neither exists yet.
	2. Build regression fixtures targeting chord-heavy passages and long staircase runs (the fuller version of the backlog item above). `test_cursor_resync_does_not_drift_over_long_progression` is a first instance, not the complete corpus.
	3. ~~Stress-test cursor resynchronization with synthetic worst-case charts.~~ **Closed 2026-07-21** — removed from the list of open architectural risks entirely, not merely "resolved." Proven property, encoded in `test_cursor_resync_does_not_drift_over_long_progression`. No further reasoning needed unless the resync mechanism itself changes.
	4. Continue documenting known trade-offs rather than overengineering fixes before there's evidence they're needed.
	**Net effect:** the project's primary outstanding work is now heuristic quality, not correctness. Every remaining open item (#1, #2, and the "does a real power chord read well" question) is a "does this produce the most readable charts?" question. More code-level correctness auditing of `mapper.py` has low expected payoff now; getting real chord-bearing `.gp` files in front of the mapper does not.
- **Timing/alignment validation (added 2026-07-22):** verify the sign and units used by `.chart` `Offset`, `song.ini` `delay`, and CLI `--offset-ms` against Clone Hero/Moonscraper behavior. Add a round-trip or fixture test once the intended convention is confirmed. The implementation should accept both positive and negative calibration values.
- **Timing/alignment validation (added 2026-07-22):** test one real song with a pickup, count-in, or meaningful leading rest. Confirm that chart tick 0 remains tied to the GP score timeline and that all event classes—notes, tempo, time signatures, and sections—share the same transform. Do not independently rebase the first guitar note.
- **Timing/alignment diagnostic rule:** after calibrating one early transient, check a late-song transient. Constant displacement means the global audio offset is wrong; growing or changing displacement means the tempo map, pickup handling, or tick conversion is wrong. A global offset must never be used to conceal drift.
