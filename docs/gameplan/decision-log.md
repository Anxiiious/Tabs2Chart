- 2026-07-20 — **7-string / Drop A open-note rule: apply to fret 0 on the single lowest-tuned string only.** `mapper._lowest_tuning_string` already infers open pitch as `pitch − fret` per string; a 7-string Drop A guitar's string 7 fret 0 naturally wins that contest. "Lowest-two-strings chug clusters" variant rejected: duplicating the rule for a second string adds complexity with zero confirmed real-file motivation, and simultaneous double-chugs are already handled by chord voicing. Reopen if real Drop A playtest shows the rule is wrong.
- 2026-07-21 — Chord interval-spread voicing (introduced 2026-07-20) **reversed and removed.** Root cause per direct user feedback: absolute-interval chord voicing was producing bad output; single-note contour anchoring was not. Removed outright rather than debugged — every note (chord members included) now goes through the same per-note lane assignment, with same-tick collisions resolved generically (nearest free lane). The 2026-07-20 open-note-rule decision's rationale referenced "the chord voicing path"; that path no longer exists, but the decision itself (fret 0 on the single lowest-tuned string only) stands unchanged.
- 2026-07-21 — Contour tracker mechanism changed from windowed min/max proportional mapping (built 2026-07-20) to a directional wraparound cursor. The windowed version flatlined long rising/falling runs at the ceiling/floor lane instead of producing the "Ladder"/staircase pattern real charts use. New mechanism: unbounded running cursor, advanced by a signed step per note-to-note interval, read out via `cursor % 5`, resetting to 0 on phrase boundaries (unchanged trigger conditions). Validated by name (Clone Hero Wiki's "Ladder" pattern) and by mechanism (Tensor Hero paper's independently-derived anchor+motion+wraparound formulation). A correctness fix, not a stylistic preference — the old version was measurably wrong for any run longer than 5 notes in one direction.
- 2026-07-21 — The `_assign_group_lanes` placeholder for a chord's non-anchor lanes resolved: replaced with a scored-candidate chord-shape heuristic (`_chord_shape_candidates` + `_rank_chord_shape`). Chosen approach: enumerate every legal placement (`C(5,k)`, ≤10) and rank by phrase-direction continuation, harmonic change/stability vs previous chord, avoiding floor/ceiling re-flattening, avoiding shape reuse from a couple chords ago, and readability (contiguous shape). Rejected alternatives: a hard adjacency rule (explicitly what the prior interval-spread voicing did, and explicitly what the collaborator handoff asked NOT to bring back); a strategy-pattern/policy-object architecture (explicitly rejected by the handoff — keep `mapper.py`'s procedural style, no DI, no plugin architecture).
- 2026-07-21 — `_rank_chord_shape`'s weights: named module-level constants, not a `ShapeWeights` config dataclass. Took the named-constants half of an external review suggestion (matches the file's existing `_RECENT_SHAPES`/`_TREND_WINDOW` convention, delivers the actual ask) but not the dataclass — crosses closer into "policy object" territory the original chord-mapper handoff explicitly asked to avoid. Revisit if a real need for multiple weight profiles (e.g. per-genre tuning) shows up.
- 2026-07-21 — Cursor-resync drift: **proven not to occur**, reclassified from "open question" to "settled property" rather than merely marked resolved. Two-part proof: (1) both the scoring-chosen and raw cursor-preferred anchor lanes are always already-wrapped into 0-4, so their difference is trivially bounded to `[-4,4]` — no unbounded quantity for error to accumulate in; (2) `_interval_to_step`'s step magnitude depends only on the real pitch interval, never the cursor's absolute value — a resync is a one-time phase-shift with no mechanism to bias any later step. Empirically confirmed over a 250-chord adversarial synthetic progression. Encoded as a permanent regression test (`test_cursor_resync_does_not_drift_over_long_progression`) rather than left as a one-off finding. Per reviewer feedback, deliberately *not* filed as "resolved" — removed from the list of open architectural risks entirely; any future change reintroducing real drift will be caught by the test, not by re-derivation of this argument.
- 2026-07-22 — **Formal architecture decision (documentation cleanup pass):** direct GPIF parsing is the primary Stage 1 architecture for GP7/8 `.gp` files, superseding Route A (external Guitar Pro/TuxGuitar conversion) as the prototype path. Recorded now for the record: (1) Route A is superseded, not merely deprioritized — no code path in the repo performs external .gpx→.gp5 conversion, and none is planned; (2) direct GPIF parsing (`gpx_reader.py` + `gpif_tempo.py`/`ir_gpif.py`) is the primary architecture for real-world input, since every real Sheet Happens file encountered so far is GP7 zip format; (3) PyGuitarPro (`tempo.py`/`ir_gp.py`) is retained, unreversed, as the parser for legacy `.gp3`/`.gp4`/`.gp5` files — a genuinely different use case from Route A, not a vestige of it; (4) the GP6 `.gpx` BCFS/BCFZ reader is retained for compatibility per the existing 2026-07-20 decision (unchanged, cross-referenced here for completeness), pending a full library audit that hasn't happened. Architecture §3 has been rewritten to state this as the current implementation, with Route A/B kept only as historical context. No behavior changed by this entry — it documents a fact about the code that was already true.
